# Human Resources (App 205) — Requirements & Feature Assessment

**Purpose:** Master-data backbone for people: employee directory, organization structure, job architecture (families → jobs → positions), grades, employment contracts, salary history, and employee documents. Other modules (DT per-diem by grade, PC custodians, FL contract owners) consume HR's data.

**Primary sources:** `final apps/HR/STATUS.md`, `final apps/HR/db/01_hr_ddl.sql` (8 tables) + `02_hr_dct_alterations.sql`, `final apps/HR/db/05_hr_pkg.sql`, `final apps/HR/db/06_hr_ords.sql` (**live** at `/ords/admin/hr/`), `final apps/HR/Jet/` (13 views, live ORDS).

---

## 1. Actors

| Actor | Capabilities |
|---|---|
| HR Admin (`IS_HR_ADMIN`) | Full CRUD on all HR masters |
| HR Manager (`IS_HR_MANAGER`) | Employee/assignment management |
| Module consumers | Read employee/grade/org data via shared tables |

## 2. Functional requirements

### 2.1 Employee directory (implemented)
- Employee master extends shared `dct_employees` (person_id PK, manager self-reference, `sync_source` HR_SYSTEM | MANUAL | API); HR added `grade_id` FK via `02_hr_dct_alterations.sql`.
- Photo upload: `PHOTO_URL` stores a relative ORDS path served via `/ords/admin/hr/employees/:id/photo` (`final apps/HR/STATUS.md` gotchas).
- JET: `employees.html`, `employeeDetail.html`.

### 2.2 Organization structure (implemented)
- Org chart over `dct_organizations` adjacency list (`parent_org_id`), recursive CTE for the tree; HR module redesigned the tree UI (commit 4563fc4).
- JET: `orgHierarchy.html`.

### 2.3 Job architecture (implemented)
- `hr_job_families` → `hr_jobs` (grade range, required qualification via lookup, min experience, CLOB responsibilities, date-effective) → `hr_positions` (budgeted headcount slots: job + org + location + grade, approved_headcount, date-effective).
- JET: `jobs.html`, `positions.html`, `grades.html`.

### 2.4 Grades (implemented)
- Shared `dct_employee_grades` (E/P/T/S categories, salary bands); grade category moved from CHECK constraint to `DCT_LOOKUP_VALUES` (`LOOKUP_TYPE = 'GRADE_CATEGORY'`) — a precedent for the lookup-migration recommendation in the DB proposal.

### 2.5 Locations (implemented)
- Physical sites: country/emirate/city/area, building, floor, Google Maps URL, type via lookup — `hr_locations`. JET: `locations.html`.

### 2.6 Contracts, salary, documents (implemented at DB level)
- `hr_employment_contracts` (type, dates, salary, currency), `hr_salary_history` (effective-dated transaction log), `hr_employee_documents` (type FK, expiry + alert lead-time).
- Document expiry navigation exists in the JET app (commit 4563fc4 "doc expiry nav"). JET: `documents.html`.

### 2.7 Assignments (implemented at DB level; **thin UI**)
- `hr_employee_assignments`: position + grade + org + location snapshot, employment_type (PERMANENT | TEMPORARY | CONTRACTOR | EXTERNAL). No dedicated JET view — assignments are only touched through employee detail.

### 2.8 Inferred requirements (not written down)
- HR is the de-facto **system of record for `dct_employees`**, yet `sync_source` implies an external HR-system feed that has no documented integration spec.
- Salary data lives in HR tables with no row-level security beyond role checks — a confidentiality requirement that should be made explicit (VPD/column masking decision).
