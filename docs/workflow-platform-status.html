The engine decision

The brief was to assess whether the Oracle APEX Workflow engine could carry these requirements. It cannot — for two reasons found in the database itself, not in the documentation. Everything is still delivered, by our own metadata.

Requirement APEX DWP (native) Why
1 · Edit approval chains from end-user UI No Yes APEX has no runtime API to create or modify a workflow definition — they are App Builder metadata.
2 · Custom outcomes (Endorse, Agree, FYI…) No Yes Every APEX outcome column is 8 characters, restricted to APPROVED/REJECTED.
3 · Customisable bilingual notifications Partial Yes Templates with EN+AR mandatory at publish — today no caller populates the Arabic columns at all.
4 · One worklist across all modules Partial Yes APEX’s task list lives inside APEX; the real UI is the JET SPAs.
5 · Dynamic approval levels from conditions No Yes Conditions must be authorable at runtime; APEX fixes them at definition time.

Blocker 1 — custom outcomes cannot be stored

WWV_FLOW_TASKS$.OUTCOME is VARCHAR2(8) with CHECK (outcome IN ('APPROVED','REJECTED')). There is nowhere to put ENDORSE or RETURN_FOR_INFO. The API’s t_task_outcome VARCHAR2(32) subtype is a red herring — the parameter is 32 chars; the storage is 8 and constrained.

Blocker 2 — definitions are not runtime-editable

APEX_WORKFLOW exposes only start / terminate / suspend / resume / retry. Changing a chain would mean an application export and deploy.

Migration sequence

Strangler-fig: both engines run side by side behind one unified inbox, so users never see two worklists. Simplest chains first; the newest and most visible chain (the Freelancer termsheet, live only since 14 Jul) goes last. Cutover and rollback are each a single UPDATE to DCT_WF_ROUTE.

PhaseRequest typesDrain capRationale
0 done Foundation — no behaviour change — 24 tables, outcome vocabulary, condition evaluator. All 13 modules stay on LEGACY.
1 FL_REGISTRATION pilot 30 d Shortest chain, one callback, lowest blast radius.
2 FL_PROFILE_CHANGE 30 d Single step, same package — doubles the pilot’s evidence for free.
3 CC_REQUEST, CC_REPLENISHMENT 45 d Retires the smallest of the three private engines.
4 PETTY_CASH, REIMBURSEMENT, CLEARING 60 d Retires the inline ORDS chain. Month-end and quarter-end must both pass.
5 TRAVEL_REQUEST, SETTLEMENT 60 d Simplest chains — deliberately late, as the safest thing to leave alone.
6 FL_CONTRACT, AMENDMENT, RENEWAL, VOUCHER 45 d Last. The 7-step termsheet chain has not completed one production cycle yet.
